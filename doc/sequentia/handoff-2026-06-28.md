## A. Consensus: anchor reorg-of-reorg recovery (SHIPPED)

Repo `Sequentia`, commit `23b1be0b1`. Design + rationale:
`doc/sequentia/anchor-reorg-of-reorg-recovery-design.md`. Test:
`test/functional/feature_pos_reorg_of_reorg_recovery.py` (passes; asserts recovery across a
restart restores the original blocks verbatim, AND that a manual `invalidateblock` is NOT
resurrected across a restart).

The three shipped changes (all node-local, NO consensus-rule change, so old/new binaries are
consensus-compatible):
1. **Seed the recovery worklist at startup** (`SeedAnchorInvalidated` ← `LoadBlockIndex`) so the
   anchor watcher's recovery survives a restart.
2. **Dedicated provenance marker** `BLOCK_FAILED_ANCHOR = 512` (chain.h, OUTSIDE
   `BLOCK_FAILED_MASK`) set ONLY by the watcher (`CChainState::MarkAnchorInvalid`), cleared in
   `ResetBlockFailureFlags`. Distinguishes anchor invalidations from `invalidateblock`/consensus
   failures so those stay durably invalid across restart (the first adversarial review caught
   that inferring provenance from `BLOCK_FAILED_VALID` alone was a deploy-blocker).
3. **Negative anchor cache** (`g_anchor_stale_cache`, cleared on parent tip change) so the
   every-tick reconsider loop RPC-checks each orphaned anchor at most once per parent-tip epoch
   (no self-inflicted bitcoind RPC storm); plus break-on-NO_CONNECTION and a once-per-batch
   `RecalculateBestHeader`.

Files touched: `src/chain.h`, `src/anchor.{h,cpp}`, `src/validation.{h,cpp}`,
`src/node/blockstorage.cpp`.

**Change 4 — NEEDS ALBERTO (not implemented).** Committee equivocation: two different blocks at
the SAME height both anchored to the SAME (valid) Bitcoin block reach quorum among disjoint
committee subsets; the minority then can't reorg to the longer chain (immediate finality), and
anchoring can't break the tie (both anchors OK). Operational band-aid = `invalidateblock` the
stuck minority tip (preserved across restart by the marker above). Real fix = a consensus
tie-break rule (yield finality to a same-anchor, strictly-better-certified sibling). Open
questions are in the design doc and the Alberto PDF. NOTE: the live 96/4 split this session
resolved itself when a deep testnet4 reorg rolled the chain back below the split — but the gap
remains.

## B. Order-book DEX (#6) — THE MAIN DIRECTIVE (resume here)

Goal: turn the web-wallet Swap tab into a real P2P **order-book** DEX (not the RFQ model), for
BOTH same-chain atomic swaps AND cross-chain BTC↔asset HTLC swaps. The RFQ
(`seqdexd`) was only ever to bootstrap a market-maker; the order book is the real product.

Repo: **`~/seqdex`** (Go, branch **`main`** — work goes directly on main).
Design docs (in `Sequentia/doc/sequentia/`): `seqob-orderbook-design.md`,
`cross-chain-orderbook-consolidation.md`.

Architecture:
- **Relay** `cmd/seqobd` — settlement-AGNOSTIC: couriers OPAQUE E2E-sealed swap messages by
  session_id and serves resting SIGNED offers; never parses settlement. E2E `Crypter` = ECDH
  secp256k1 → AES-256-GCM (`internal/seqob/client/crypter.go`).
- **Maker** `cmd/seqob-maker` (our MM; anyone can make the same way). `-mode samechain|cross`.
- **Taker** `cmd/seqob-cli`.
- Proto: `daemon/api-spec/protobuf/seqob/v1/{offer,relay}.proto`. Offer `Settlement` oneof
  `{SameChainTerms=20, CrossChainTerms=21, LightningTerms=22}`.

  binaries in `/root/seqob-test/`, Restart=always) — see memory `seqob-box-runtime.md`.
